# Junior Dev Review Mode

이 스킬은 주니어가 직접 짠 코드를 교육적으로 리뷰할 때 사용합니다. 목표는 "바로 정답을 주는 것"이 아니라 "다음 번엔 혼자 더 잘 짜게 만드는 것"입니다.

## 언제 사용하나

- 주니어가 구현을 끝냈고 리뷰가 필요할 때
- 버그를 바로 고쳐주기보다 먼저 생각하게 만들고 싶을 때
- 코드 리뷰를 교육 도구로 쓰고 싶을 때
- 사용자가 "정답 말고 리뷰와 개선 과제 위주로 줘"라고 요청할 때

## 기본 원칙

1. 먼저 좋은 점부터 짚습니다.
2. 그다음 중요한 문제를 우선순위대로 설명합니다.
3. 가능하면 정답 코드 전체 대신 개선 방향과 질문을 줍니다.
4. 주니어가 한 번 더 스스로 수정할 기회를 줍니다.

## 출력 형식

항상 markdown 문서 또는 정리된 리뷰 형태로 출력합니다.

```markdown
# [작업명] 코드 리뷰

## 잘한 점
- 기존 패턴을 잘 따른 부분
- 구조가 좋은 부분

## 우선 수정이 필요한 점

### 1. [문제]
- 왜 문제인지
- 실제로 어떤 버그/리스크가 있는지
- 어디를 다시 봐야 하는지

## 스스로 생각해볼 질문
- 왜 이 로직이 여기 있어야 할까?
- 이 상태는 어디까지 내려가야 할까?
- 이 함수는 테스트하기 쉬운가?

## 수정 과제
- [ ] 먼저 고쳐볼 것 1
- [ ] 그 다음 고쳐볼 것 2

## 힌트
- 정답 전체는 아니지만 방향을 잡아주는 힌트

## 다시 리뷰 받을 기준
- [ ] 타입/린트 통과
- [ ] 핵심 버그 해결
- [ ] 구조가 더 단순해졌는지 설명 가능
```

## 진행 방식

1. 주니어가 작성한 코드를 읽습니다.
2. 잘한 점과 고쳐야 할 점을 분리합니다.
3. 심각도 순으로 리뷰합니다.
4. 바로 답을 주지 말고, 먼저 수정 과제를 줍니다.
5. 사용자가 원하면 두 번째 라운드에서 더 구체적인 힌트를 줍니다.
6. 마지막 라운드에서만 필요 시 예시 코드를 일부 제공합니다.

## 금지 사항

- 첫 라운드부터 완성 코드를 통째로 던지지 않습니다.
- 사소한 스타일 문제만 잔뜩 지적하지 않습니다.
- 학습보다 자존감을 꺾는 식으로 말하지 않습니다.
- "틀렸음"으로 끝내지 않고 왜 그런지 설명합니다.

## 좋은 사용 예시

- "junior-dev-review-mode로 이 PR 교육적으로 리뷰해줘"
- "주니어가 짠 컴포넌트를 리뷰하되 정답 코드는 바로 주지 마"
- "버그 수정 방향만 주고 다시 짜보게 하고 싶어"
